# Research Brief: bdd-suite

**Date:** 2026-04-23
**Agent:** research-first
**PRD:** docs/prds/bdd-suite.md
**Feature:** pure cucumber-js + Playwright library + TS step defs (Feature 1 of a three-feature plan)

## Summary

- **7 libraries researched** in detail (primary) + **3 secondary topics** summarised.
- **Key discovery 1 — `ts-node` is no longer the cucumber-recommended TS loader.** As of cucumber-js 12.x, the project documentation recommends `tsx` (not `ts-node`) for on-the-fly TypeScript compilation. `ts-node/register` still works for CommonJS, but given the frontend's `"type": "module"`, we should scaffold with `tsx/esm` from day one. The PRD's "`ts-node/register` via `requireModule`" wording is stale.
- **Key discovery 2 — The legacy `json` formatter is in maintenance mode.** cucumber-js docs say "this formatter is in maintenance mode and won't have new features added." New consumers are pointed at the `message` (NDJSON) formatter + the standalone `@cucumber/json-formatter` converter. This directly affects Feature 2's ingestion path.
- **Key discovery 3 — cucumber-js 12 requires Node 20+ (dropped 18 and 23 in 12.0.0).** We're on Node 22 so we're fine, but the pin matters for the `engines` field.

## Libraries

### `@cucumber/cucumber`

- **Current stable version (April 2026):** `12.8.1` (released 2026-04-14).
- **Recommended pin:** `^12.8.1` — caret is safe because 12.x has been running since July 2025 without a 13.x on the horizon. Pin major in `engines`.
- **Sources (accessed 2026-04-23):**
  - [Releases · cucumber/cucumber-js (GitHub)](https://github.com/cucumber/cucumber-js/releases) — accessed 2026-04-23
  - [@cucumber/cucumber (npm)](https://www.npmjs.com/package/@cucumber/cucumber) — accessed 2026-04-23
  - [cucumber-js CHANGELOG.md (raw)](https://raw.githubusercontent.com/cucumber/cucumber-js/main/CHANGELOG.md) — accessed 2026-04-23
- **Key patterns / current idiom:**
  - Config file supports `cucumber.json|yaml|yml|js|cjs|mjs|ts|mts|cts` — resolved in that order at project root. For an ESM project with `"type": "module"`, `cucumber.mjs` or `cucumber.cjs` (explicit) are both valid; `cucumber.cjs` is simplest because it avoids the ESM default-export dance.
  - Programmatic option keys (authoritative list): `paths`, `format`, `formatOptions`, `import`, `require`, `requireModule`, `tags`, `worldParameters`, `parallel`, `retry`, `strict`, `publish`, `publishQuiet` (removed in 12.0).
  - `--tags` accepts a boolean infix expression: `"@smoke and not @wip"`, `"@happy or @edge"`. Feature-level tags are inherited by all scenarios; Scenario Outline tags are inherited by Examples rows.
  - Step/hook APIs: `Given/When/Then`, `Before`, `After`, `BeforeAll`, `AfterAll`, `BeforeStep`, `AfterStep`. After-hooks fire in **reverse** registration order. Tag-filter shorthand: `Before("@smoke", fn)` or `Before({tags: "@smoke and @critical"}, fn)`.
- **Pitfalls / breaking changes since 2024:**
  - **v11.0.0 (Aug 2024):** removed deprecated `parseGherkinMessageStream`, `PickleFilter`, `Runtime` APIs; dropped Node 21.
  - **v12.0.0 (Jul 2025):** **dropped Node 18.x and 23.x** (requires ≥ 20); removed `publishQuiet` option; redesigned HTML formatter header.
  - **v12.4.0 (Dec 2025):** added TypeScript config file support (`cucumber.ts`).
  - **v12.7.0 (Feb 2026):** fixed ESM source references + env-var propagation in parallel mode.
  - Arrow functions in hooks/steps do **not** bind `this` to the World — use `function()` or accept the World as `this: CustomWorld` parameter.
- **Design impact:** We pin `^12.8.1`, scaffold `frontend/cucumber.cjs` (simplest for ESM package), and wire tag expressions into `pnpm bdd -- --tags "@smoke and not @wip"`. Node engine requirement in `frontend/package.json` should read `"engines": { "node": ">=20" }`.
- **Test implication:** Scenarios MUST avoid arrow functions in step-def bodies that need World access; the lint check in Feature 2 can assert this. Tag inheritance (feature-level `@smoke` propagates) is load-bearing — docs must teach it because the PRD explicitly allows feature-level `@smoke`.

### `playwright` (library, not `@playwright/test`)

- **Current stable version (April 2026):** `1.59.1` (released early April 2026); `1.60.0-alpha-2026-04-18` exists but not stable.
- **Recommended pin:** `^1.59.0` — match the existing `@playwright/test` pin already in `package.json` (line 23). Keep both libs in lockstep to avoid browser-version drift.
- **Sources (accessed 2026-04-23):**
  - [playwright (npm)](https://www.npmjs.com/package/playwright) — accessed 2026-04-23
  - [Playwright Library docs](https://playwright.dev/docs/library) — accessed 2026-04-23
  - [APIRequestContext docs](https://playwright.dev/docs/api/class-apirequestcontext) — accessed 2026-04-23
  - [Playwright release notes](https://playwright.dev/docs/release-notes) — accessed 2026-04-23
- **Key patterns / current idiom:**
  - Browser lifecycle (library mode):
    ```ts
    import {
      chromium,
      request,
      Browser,
      BrowserContext,
      Page,
      APIRequestContext,
    } from "playwright";
    const browser: Browser = await chromium.launch();
    const context: BrowserContext = await browser.newContext();
    const page: Page = await context.newPage();
    // ... scenario work ...
    await context.close();
    await browser.close();
    ```
  - HTTP from step defs without `node-fetch`: `const api: APIRequestContext = await request.newContext({ baseURL: "http://localhost:8000" });` then `api.get(...)`, `api.post(...)`, `api.dispose()`. Shares no cookies with the browser context unless you explicitly re-use `context.request`.
  - Per-scenario `BrowserContext` + shared `Browser` is the canonical isolation pattern in cucumber-js BDD suites (Tallyb/cucumber-playwright, ediblecode blog).
- **Pitfalls / breaking changes since 1.50:**
  - **v1.49:** Chrome/Edge channels switched to new headless mode (check CI expectations — not applicable here, local-only).
  - **v1.52:** glob URL patterns dropped `?` and `[]`; `route.continue()` no longer overrides `Cookie`.
  - **v1.58:** removed `_react`/`_vue` selectors; removed `:light` selector engine; removed `devtools` option from `browserType.launch()`.
  - **v1.59:** removed macOS 14 WebKit support and `@playwright/experimental-ct-svelte`.
  - Note: `page.accessibility` deprecated since 1.57. Not used in our PRD scenarios.
- **Design impact:** We import from `"playwright"` (not `"@playwright/test"`). The existing `@playwright/test@^1.59` dependency's already-installed browsers are reused via `node_modules/playwright-core` — no second `pnpm exec playwright install` needed. `APIRequestContext` via `request.newContext({ baseURL })` replaces the PRD's "Node's `fetch` or `node-fetch`" hand-wave and avoids adding a dependency.
- **Test implication:** Each scenario MUST close its `BrowserContext` in the `After` hook; leaking contexts between scenarios breaks the isolation contract the PRD asserts. Tests should cover: context-per-scenario, browser-shared-across-suite, `AfterAll` closes browser cleanly on both pass and failure paths.

### `ts-node` / `tsx` (TypeScript runtime loader)

- **Current stable version (April 2026):** `ts-node` 10.9.x (last release summer 2024, now in maintenance). `tsx` is actively maintained; latest ~4.19.x.
- **Recommended pin:** **Use `tsx`, not `ts-node`.** Install `tsx` as a dev-dependency; do not install `ts-node` at all. Pin `~4.19`.
- **Sources (accessed 2026-04-23):**
  - [cucumber-js transpiling.md (main branch)](https://github.com/cucumber/cucumber-js/blob/main/docs/transpiling.md) — accessed 2026-04-23
  - [State of TypeScript 2026](https://devnewsletter.com/p/state-of-typescript-2026/) — accessed 2026-04-23
  - [ts-node (npm)](https://www.npmjs.com/package/ts-node) — accessed 2026-04-23
- **Key patterns / current idiom:**
  - cucumber-js docs (main, 2025-2026) recommend **tsx** as the just-in-time TS compiler. For ESM (our case):
    ```js
    // cucumber.cjs
    module.exports = {
      default: {
        import: ["tsx/esm", "tests/bdd/steps/**/*.ts"],
        paths: ["tests/bdd/features/**/*.feature"],
        format: ["progress-bar", "json:test-results/cucumber.json"],
      },
    };
    ```
  - For CommonJS (not our case): `requireModule: ["tsx/cjs"], require: ["…/*.ts"]`.
  - `tsx` does not need `tsconfig.json` to run, but honors `moduleResolution: "Bundler"` or `"NodeNext"` when one exists.
- **Pitfalls / breaking changes since 2024:**
  - **Major ecosystem shift:** Node 22.18+ ships native TypeScript execution; community guidance is to drop `ts-node`/`tsx` dependencies after late 2026. For April 2026, `tsx` is still the stable choice; native Node TS is not yet the default.
  - `ts-node/esm` + cucumber-js has a known bug: TypeScript path aliases (`@/…`) fail to resolve in ESM mode (cucumber-js issue #2403). `tsx` handles this correctly.
  - `ts-node` is effectively unmaintained; no releases in ~12 months as of today.
- **Design impact:** PRD line 95 says `requireModule: ts-node/register` — this is stale and must be updated in the plan. The `frontend/cucumber.cjs` should use `import: ["tsx/esm", …]`. We add `tsx` to `frontend/devDependencies`; we do NOT add `ts-node`.
- **Test implication:** The BDD config must be validated by actually running `pnpm bdd --dry-run` as part of setup; a "TS step-def loads without transpile error" test is essential because the ESM+TS interop is where most cucumber+TS projects stumble.

### Cucumber step-definition TypeScript signatures + custom World

- **Current stable version (April 2026):** ships as part of `@cucumber/cucumber` 12.8.1; types are bundled (no `@types/cucumber` needed; that package is legacy).
- **Recommended pin:** covered by `@cucumber/cucumber`.
- **Sources (accessed 2026-04-23):**
  - [cucumber-js world.md docs](https://github.com/cucumber/cucumber-js/blob/main/docs/support_files/world.md) — accessed 2026-04-23
  - [Tallyb/cucumber-playwright `custom-world.ts`](https://github.com/Tallyb/cucumber-playwright/blob/master/src/support/custom-world.ts) — accessed 2026-04-23
  - [Dilshan Kelsen — Custom World for Cucumber/Playwright/TS](https://dilshankelsen.com/set-up-custom-world-for-cucumber-playwright-tests-in-typescript/) — accessed 2026-04-23
- **Key patterns / current idiom:**
  - Import surface: `import { Given, When, Then, Before, After, BeforeAll, AfterAll, World, setWorldConstructor, IWorldOptions } from "@cucumber/cucumber";`
  - Custom World (our shape):

    ```ts
    import {
      World,
      IWorldOptions,
      setWorldConstructor,
    } from "@cucumber/cucumber";
    import {
      Browser,
      BrowserContext,
      Page,
      APIRequestContext,
    } from "playwright";

    export interface HangmanWorldParams {
      backendBaseUrl: string;
      frontendBaseUrl: string;
    }

    export class HangmanWorld extends World<HangmanWorldParams> {
      browser!: Browser;
      context!: BrowserContext;
      page!: Page;
      api!: APIRequestContext;
      constructor(options: IWorldOptions<HangmanWorldParams>) {
        super(options);
      }
    }
    setWorldConstructor(HangmanWorld);
    ```

  - Steps access World via `this` when declared as `function`, not arrow.

- **Pitfalls:**
  - Arrow functions in steps break `this` binding.
  - Non-null assertions (`browser!`) are the standard pattern because `BeforeAll` populates them post-construction.
- **Design impact:** Plan must specify a `HangmanWorld` class (recommended filename `tests/bdd/support/world.ts`) and a `tests/bdd/support/hooks.ts` that uses `BeforeAll`/`Before`/`After`/`AfterAll` to manage browser lifecycle. `worldParameters` (from config or CLI) is the clean path to pass `backendBaseUrl`/`frontendBaseUrl`.
- **Test implication:** Add a smoke step that asserts `this.api` and `this.page` are both defined — catches misordered hook registration early.

### Cucumber JSON reporter format

- **Current stable version (April 2026):** the **legacy** `json` formatter ships with `@cucumber/cucumber` 12.8.1 and continues to work. The modern alternative is `message` (NDJSON) + `@cucumber/json-formatter` CLI converter.
- **Recommended pin:** ship **both** formatters for Feature 2's optionality:
  ```
  format: [
    "progress-bar",
    "json:test-results/cucumber.json",        // legacy — familiar, widely parsed
    "message:test-results/cucumber.ndjson"     // modern — stable schema going forward
  ]
  ```
- **Sources (accessed 2026-04-23):**
  - [cucumber-js formatters.md](https://github.com/cucumber/cucumber-js/blob/main/docs/formatters.md) — accessed 2026-04-23
  - [cucumber/cucumber-json-schema (GitHub)](https://github.com/cucumber/cucumber-json-schema) — accessed 2026-04-23
  - [JsonFormatter API reference](<https://cucumber.github.io/cucumber-js/classes/(root).JsonFormatter.html>) — accessed 2026-04-23
- **Key patterns / current idiom:**
  - Legacy JSON schema (array of feature objects):
    - Feature: `line`, `uri`, `id`, `keyword`, `name`, `description`, `elements[]`, `tags[]`
    - Scenario (`element`): `start_timestamp`, `line`, `id`, `type`, `keyword`, `name`, `description`, `steps[]`, `tags[]`
    - Step: `keyword`, `line`, `match{location, arguments[]}`, `name`, `result{duration, status}`
  - Status enum: `passed`, `failed`, `skipped`, `undefined`, `pending`, `ambiguous`.
  - Message format: newline-delimited JSON, one message per line (testRunStarted, pickle, testCaseStarted, testStepFinished, …). Better for streaming and tool interop.
- **Pitfalls / breaking changes:**
  - Legacy JSON **won't get new features** — anything added to Cucumber (e.g., new hook types' attribution) may not surface.
  - Standalone-json-formatter issue #8: regular steps can end up under `after` section (a known data-shape gotcha Feature 2 must handle).
- **Design impact:** Emit BOTH files. Feature 2 can start with `cucumber.json` (simple, widely documented) and later switch to parsing `cucumber.ndjson` without a Feature 1 rewrite.
- **Test implication:** Add a smoke check in `make bdd` that both files exist post-run and parse as valid JSON / NDJSON. If we only emit one and Feature 2 later wants the other, we ship-retrofit.

### Cucumber tag expressions

- **Current stable version (April 2026):** part of `@cucumber/cucumber` 12.8.1; the Cucumber tag-expressions DSL has been stable for 5+ years.
- **Recommended pin:** covered by `@cucumber/cucumber`.
- **Sources (accessed 2026-04-23):**
  - [Cucumber tag reference](https://cucumber.io/docs/cucumber/api/) — accessed 2026-04-23
  - [cucumber-js issue #469 — tag syntax](https://github.com/cucumber/cucumber-js/issues/469) — accessed 2026-04-23
- **Key patterns / current idiom:**
  - Operators: `and`, `or`, `not`, parentheses.
  - Examples: `--tags "@smoke"`, `--tags "@happy and not @wip"`, `--tags "(@happy or @edge) and @smoke"`.
  - Inheritance: tags above `Feature` propagate to all `Scenario`/`Scenario Outline`; tags above `Scenario Outline` propagate to each row in `Examples`.
  - A single scenario can carry multiple tags on the same line: `@happy @smoke`.
- **Pitfalls:**
  - No change in syntax — the old `~@tag` negation (pre-2017) is gone; must use `not @tag`.
  - Shell quoting: always wrap multi-word expressions in double quotes.
- **Design impact:** `make bdd` can accept a pass-through flag: `make bdd TAGS='@smoke'` → `pnpm bdd -- --tags "$TAGS"`. The PRD's acceptance criterion "a scenario is either @happy, @failure, or @edge — never multiple" is a style rule, not enforced by cucumber — Feature 2's lint enforces it.
- **Test implication:** Add three smoke checks: (1) `--tags @smoke` filters correctly; (2) feature-level tag propagates; (3) `not` operator works in shell-quoted form.

## Cross-cutting findings

### cucumber-js + tsx + ESM/CJS interop

- **The frontend is ESM** (`"type": "module"`, Vite 8, Node 22). Any TS loader must honor this.
- **Choose `tsx` over `ts-node`.** cucumber-js docs (main branch, 2025+) recommend it; `ts-node` is in maintenance; the path-alias bug in ts-node+ESM (cucumber-js #2403) is a real footgun.
- **Config file choice:** `frontend/cucumber.cjs` is simplest — it can use `module.exports =` regardless of the package.json `"type": "module"`. `cucumber.mjs` / `cucumber.ts` also work but add ceremony.
- **TS compile target:** add `"module": "NodeNext"`, `"moduleResolution": "NodeNext"`, `"target": "ES2022"` in a dedicated `tests/bdd/tsconfig.json` (extends root) to avoid conflicts with Vite's `"module": "ESNext"` expectations. `tsx` does not require strict tsconfig settings but respects them.

### Cucumber JSON reporter schema stability

- **Legacy JSON schema** (the one `--format json:…` produces) has been stable since ~2019 and is documented by the community in `cucumber/cucumber-json-schema`. It is **frozen in maintenance mode**, which paradoxically is good for stability but bad for long-term features.
- **Modern message format (NDJSON)** is the forward-looking schema; it's versioned via `@cucumber/messages` (separate npm package). Feature 2 should probably target NDJSON.
- **Recommendation:** emit both. Zero cost, full optionality.

### Node.js + browser installation

- cucumber-js 12.x requires **Node ≥ 20**; we're on Node 22 (lockfile).
- Playwright 1.59 browsers are already installed via `@playwright/test`; `playwright` library reuses `playwright-core` from the same `node_modules` — no second `playwright install`.

## Open Risks

- **Risk 1: `tsx` version pin churn.** tsx ships frequently (major every ~6 mo historically). Pin with `~4.19` and budget a check on next upgrade.
- **Risk 2: Native Node TS adoption in late 2026.** Node 22.18 has native TS, and community guidance post-Oct 2026 is to drop `tsx`. For this PR we stay on `tsx`; a follow-up ticket in Q4 2026 may migrate.
- **Risk 3: Legacy JSON formatter quirks.** Issue #8 in `cucumber/json-formatter` shows edge cases where step arrays can land in unexpected keys. Feature 2's parser must tolerate this. Documented, not blocking F1.
- **Risk 4: Playwright 1.60 is on track.** Alpha exists (`1.60.0-alpha-2026-04-18`). If it ships before the plan is implemented, we stay on 1.59 to match the already-installed `@playwright/test` — don't split the browsers.
- **Risk 5: `worldParameters` vs env vars.** PRD says step defs read `HANGMAN_BACKEND_PORT` / `HANGMAN_FRONTEND_PORT`. Cucumber's canonical path is `--world-parameters '{"backendBaseUrl":"http://localhost:8000"}'`. Plan should resolve: prefer `worldParameters` with env-var fallback in the World constructor.

## Not Researched (explicit triage)

- **`playwright-bdd`** — explicitly rejected in PRD Q1 (documentation purity goal). One-line triage: not on the shelf.
- **`@playwright/test` fixtures** — stays installed (existing spec files) but the BDD suite does NOT use its fixtures, `webServer`, or test runner. Browsers from its install are reused.
- **`cucumber-html-reporter`, `@cucumber/pretty-formatter` (external package)** — out of F1 scope; F2 generates the dashboard. Built-in `progress-bar` / `summary` formatters are sufficient for F1.
- **Parallelism (`--parallel N`)** — PRD locks to default 1. Single-user scale.
- **`behave` / `pytest-bdd`** — Python BDD frameworks, not applicable (JS stack chosen).
- **`@types/cucumber`** — legacy package; types ship inside `@cucumber/cucumber` since v7. Do NOT install.
- **`node-fetch`** — unnecessary; Playwright's `APIRequestContext` plus Node 22's built-in `fetch` cover all HTTP calls.
- **CI config** — PRD explicitly local-only.
- **Auth / cookies for test** — hangman has no auth; the session-cookie flow is part of the app surface under test, not framework plumbing.
